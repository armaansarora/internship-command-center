## 1. Animation Layer: GSAP + Framer Motion

### GSAP v3.13 (All Plugins Free)
As of early 2025, GSAP made all premium plugins permanently free for all use cases, including commercial. This is a massive unlock for The Tower.

**Plugins we use:**
| Plugin | Use Case | Notes |
|---|---|---|
| ScrollTrigger | Parallax skyline, scroll-driven floor reveals | Pin sections, scrub timelines |
| SplitText | Character name reveals, heading animations | Split by chars/words/lines |
| Flip | Application status changes (Kanban drag) | Animate layout shifts smoothly |
| DrawSVG | Skyline building outlines at night | Animate SVG stroke paths |
| MorphSVG | Day/night sky transitions | Morph between sun and moon shapes |
| MotionPath | Elevator indicator movement, cursor trails | Animate along bezier paths |
| Inertia | Drag interactions (Kanban cards, skyline scroll) | Physics-based throw |
| Physics2D | Particle effects (dust motes, city lights) | Optional ambient effects |

**Implementation:**
```bash
npm install gsap
```
```js
import { gsap } from 'gsap';
import { ScrollTrigger } from 'gsap/ScrollTrigger';
import { SplitText } from 'gsap/SplitText';
import { Flip } from 'gsap/Flip';
gsap.registerPlugin(ScrollTrigger, SplitText, Flip);
```
Tree-shake by importing only used plugins. No license key needed.

Sources: [CSS-Tricks](https://css-tricks.com/gsap-is-now-completely-free-even-for-commercial-use/), [Codrops demos](https://tympanus.net/codrops/2025/05/14/from-splittext-to-morphsvg-5-creative-demos-using-free-gsap-plugins/)

### Framer Motion (React Integration)
Use for: component mount/unmount, layout animations, hover states, shared layout transitions.

**Coexistence rule:** GSAP owns the timeline (elevator, scroll, complex sequences). Framer Motion owns React lifecycle (page enter/exit, conditional renders, drag). Never animate the same property with both libraries simultaneously.

Source: [Reddit r/reactjs discussion](https://www.reddit.com/r/reactjs/comments/txi62v/is_it_bad_practice_to_use_gsap_and_framer_motion/)

## 2. Procedural NYC Skyline

### Approach: Layered SVG with Trig Functions
Based on the kevinverse technique, adapted for a cityscape instead of landscapes.

**How it works:**
1. Multiple SVG `<path>` elements, each representing a depth layer
2. Height values generated by interfering (adding) sine waves of different frequencies
3. Each layer: different amplitude, frequency, phase, and color
4. Front layers are darker/more saturated, back layers are washed out (atmospheric perspective)
5. Sky is a `<rect>` with a radial gradient (sun position drives gradient stops)
6. Noise filter (`feTurbulence` + `feColorMatrix`) for texture

**Adaptation for cityscape:**
- Replace smooth sine curves with stepped/rectangular shapes (buildings have flat tops)
- Add window grid patterns on near-layer buildings (small bright rectangles at night)
- Use `Math.random()` seeded values for building height variation within each layer
- City lights at night: scatter `<circle>` elements with warm `fill` on dark buildings

**Day/Night integration:**
- Sky gradient stops shift based on time-of-day CSS custom property
- Building fill colors darken at night, window lights appear (opacity transition)
- Sun/moon position calculated from actual time

**Performance:**
- SVG is GPU-accelerated via `transform: translate3d(0,0,0)` on each layer
- Parallax via `will-change: transform` on mouse move
- Total SVG complexity: ~50-100 path elements (lightweight)

Source: [kevinverse procedural SVG](https://kwa.ng/procedurally-generated-svg-landscapes/)

## 3. Day/Night Cycle Engine

### CSS Custom Properties + JavaScript Clock

**7 time states:**
```
dawn:         5:00 - 7:00   → warm pink/orange sky, soft light
morning:      7:00 - 10:00  → bright, clear, energetic
midday:       10:00 - 14:00 → neutral white/blue, full brightness
afternoon:    14:00 - 17:00 → slightly warm, golden tint emerging
golden_hour:  17:00 - 19:00 → deep gold, long shadows, most luxurious
dusk:         19:00 - 21:00 → purple/orange, city lights switching on
night:        21:00 - 5:00  → dark blue/black, full city lights, stars
```

**Implementation pattern:**
```js
function getTimeState() {
  const hour = new Date().getHours() + new Date().getMinutes() / 60;
  // Returns interpolated values between adjacent states
  // CSS properties: --sky-hue, --sky-sat, --sky-light, --ambient-temp, --light-intensity
}
```

Set properties on `:root` every 60 seconds. Transitions via `transition: all 60s linear` on sky elements (imperceptible to user, smooth over time).

Source: [Stack Overflow day/night CSS](https://stackoverflow.com/questions/39572238/change-css-style-depend-on-real-day-and-night-time)

## 4. Supabase + Drizzle + RLS

### Drizzle Native RLS Support
Drizzle ORM has built-in RLS support. Define policies directly in the schema's third argument.

**⚠️ KNOWN ISSUE (as of March 2026):** `drizzle-kit` has a bug with RLS policy migration generation (GitHub issue #4198) — updating an existing policy can produce incorrect SQL. **Mitigation:** For policy changes, write raw SQL migrations instead of relying on `drizzle-kit generate`. Initial schema push (`drizzle-kit push`) works fine; the bug affects subsequent policy modifications only.

**Pattern (correct API — third-argument array, not `.withRLS()`):**
```ts
import { pgTable, uuid, text, pgPolicy } from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';

export const applications = pgTable('applications', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').notNull(),
  company: text('company').notNull(),
  status: text('status').notNull().default('saved'),
  // ...
}, (table) => [
  pgPolicy('user_isolation', {
    for: 'all',
    to: 'authenticated',
    using: sql`auth.uid() = user_id`,
    withCheck: sql`auth.uid() = user_id`,
  }),
]);
```

**Service-role bypass for background jobs (Inngest):**
```ts
// In Inngest functions, use service role client
import { createClient } from '@supabase/supabase-js';
const supabase = createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY);
// Service role bypasses RLS entirely — no SET LOCAL needed.
// Only use SET LOCAL app.user_id if you want to run queries AS a user through service role.
```

Source: [Drizzle RLS docs](https://orm.drizzle.team/docs/rls), [Drizzle GitHub #4198](https://github.com/drizzle-team/drizzle-orm/issues/4198)

### @supabase/ssr Auth Pattern

**⚠️ API UPDATE (as of 2025):** The `@supabase/ssr` package replaces the old `@supabase/auth-helpers-nextjs`. Key changes:
- Client: `createBrowserClient(url, publishableKey)` — uses `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (renamed from "anon key" in latest docs, but the actual key value is the same anon key)
- Server: `createServerClient(url, publishableKey, { cookies })` — requires cookie read/write callbacks
- Middleware: `updateSession()` pattern refreshes auth tokens on every request

**Middleware pattern (`middleware.ts`):**
```ts
import { createServerClient } from '@supabase/ssr';
import { NextResponse, type NextRequest } from 'next/server';

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      cookies: {
        getAll: () => request.cookies.getAll(),
        setAll: (cookiesToSet) => {
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          );
        },
      },
    }
  );
  const { data: { user } } = await supabase.auth.getUser();
  if (!user && !request.nextUrl.pathname.startsWith('/lobby')) {
    return NextResponse.redirect(new URL('/lobby', request.url));
  }
  return supabaseResponse;
}
```

**Data Access Layer:** Verify session server-side before any DB query (double protection beyond middleware). Wrap auth-dependent UI in `<Suspense>` for instant shell render.

Source: [Supabase SSR docs](https://supabase.com/docs/guides/auth/server-side/nextjs), [Next.js Auth Guide](https://nextjs.org/docs/app/guides/authentication)

## 5. Inngest: Background Jobs + Realtime

### Inngest Realtime API

**⚠️ STABILITY WARNING (as of March 2026):** Inngest Realtime is in **"developer preview"** — NOT production stable. APIs may change without notice. Not covered by Inngest's SLA. Safe for our use case (progress indicators are nice-to-have, not critical), but plan a fallback.

**Fallback plan:** If Realtime API breaks or changes, fall back to Supabase Realtime (Postgres LISTEN/NOTIFY) for progress streaming. The Inngest function writes progress to a `job_progress` table, and the client subscribes via Supabase's `channel.on('postgres_changes', ...)`. This is production-stable and already in our stack.

**Current Pattern (developer preview — may change):**
```ts
// In Inngest function
const { step, publish } = createFunction(...);

await step.run('process-emails', async () => {
  for (const email of emails) {
    await processEmail(email);
    await publish({ channel: `user/${userId}/sync`, data: { processed: i, total: emails.length } });
  }
});
```

```tsx
// In React component
import { useInngestRealtime } from '@inngest/react';

function SyncProgress({ userId }) {
  const { messages } = useInngestRealtime(`user/${userId}/sync`);
  return <ProgressBar value={messages.at(-1)?.processed} max={messages.at(-1)?.total} />;
}
```

**Use cases in The Tower:**
- Email sync progress (COO's room shows "Scanning inbox... 47/132")
- Agent dispatch progress ("CRO analyzing pipeline..." → "CIO researching companies..." → "Briefing compiled")
- Application import from CSV

Source: [Inngest Realtime blog](https://www.inngest.com/blog/background-jobs-realtime-nextjs)

### Inngest Function Patterns We'll Use

|---|---|
| Simple background job | Email parsing, contact sync |

## 7. Character Art Pipeline

### AI-Generated 2D Characters (Option A)

**Best pipeline for consistent illustrated characters:**
1. **Base model:** Flux.1 (open-source, high-quality)
2. **Style LoRA:** Train on 10-15 reference images in target art style (e.g., "corporate illustrated character, warm lighting, semi-realistic")
3. **Character LoRA:** Train per character on 10-15 images to lock face/body consistency
4. **ControlNet:** Use for pose control (idle at desk, standing at whiteboard, talking gesture)
5. **PuLID:** Face identity preservation across different poses/expressions

**What we need per character:**
- 3-5 pose variants: idle, talking, gesturing, working, alert
- Each pose at ~1024x1024, transparent background (or easy to mask)
- Consistent style across all 8 characters

**Alternative (faster for V1):** Use a single Flux.1 generation session with detailed prompts, manually curate for consistency, touch up in Photoshop/Figma. Less consistent but faster.

**Alternative (no AI):** Commission from Fiverr/99designs. ~$200-500 for 8 characters with 3-5 poses each. Most consistent but slowest.

## 8. Supabase Edge Functions vs. Next.js API Routes

### Decision Matrix

| Feature | Edge Functions | Next.js API Routes |
|---|---|---|
| Best for | Webhooks, DB triggers, scheduled jobs | App logic, auth-dependent routes |
| Runtime | Deno (globally distributed) | Node.js (Vercel edge or serverless) |
| Cold start | <50ms | ~200-500ms (serverless), <50ms (edge) |
| Supabase access | Direct (no network hop) | Via client library |
| Use in The Tower | Stripe webhooks, email inbound hooks | All app API routes, agent endpoints |

**Our pattern:** Next.js API routes for everything except webhooks from external services. Inngest handles all background job orchestration (not Edge Functions).

Source: [Supabase Edge Functions Guide](https://www.iloveblogs.blog/guides/nextjs-supabase-edge-functions-guide)

## 9. Visual Novel / Character Presentation References

### Pixi'VN (Web Visual Novel Engine)
A TypeScript visual novel engine built on PixiJS. Relevant patterns to steal:
- Dialogue system with character portraits and text boxes
- Character sprite layering (background → character → UI overlay)
- Dialogue animation (text reveal, character expression changes)
- Save/load state management

We don't use the engine directly — we borrow the interaction pattern for our character system.

### Rive for Character Animation (Alternative to Sprite Sheets)
Rive is a real-time animation tool that exports lightweight web animations. Could be used for:
- Character idle animations (subtle breathing, blinking, working)
- Talking animations (mouth movement, gestures)
- State transitions (idle → talking → alert)

**Pros:** Very small file size, smooth interpolation, state machine built-in.
**Cons:** Requires Rive editor for asset creation, learning curve.
**Verdict:** Consider for V1.5 or V2 upgrade. Start with CSS/GSAP sprite animations for V1.
