## Wave 2 — Migration + Parallel Dispatch Core (serial — R3.3 depends on schema existing in DB)

### R3.3 — `dispatchBatch` tool with real parallel fan-out
**Files:**
- `src/lib/ai/agents/ceo-orchestrator.ts` — add `buildDispatchBatchTool(userId, userName)` factory; export alongside existing `buildCEODispatchTools`
- `src/app/api/ceo/route.ts` — `buildExtraTools` returns `{ ...buildCEODispatchTools(...), dispatchBatch: buildDispatchBatchTool(...) }`
- `src/lib/ai/agents/ceo-orchestrator.dispatch-batch.proof.test.ts` — new (the Proof-aligned parallel test)

**Implementation:**
```ts
// pseudocode
const dispatchBatch = tool({
  description: "Dispatch 2+ department heads in parallel... Use for status-across-departments briefings.",
  inputSchema: z.object({ tasks: z.object({ cro: z.string().optional(), ...}).refine(t => count(t)>=2) }),
  execute: async ({ tasks }) => {
    const requestId = crypto.randomUUID();
    const present = Object.entries(tasks).filter(([,t]) => !!t);
    const ids = await Promise.all(present.map(([agent, task]) =>
      insertQueuedDispatch(userId, requestId, agent, task!, [])));
    const results = await Promise.allSettled(
      present.map(([agent, task], i) => runSubagentWithDispatchId(ids[i], {...}, agent, task!))
    );
    return { requestId, agents: results.map(...) };
  }
});
```

`runSubagentWithDispatchId` wraps existing `runSubagent`: calls `markDispatchRunning` before `generateText`, `completeDispatch` on success, `failDispatch` on catch.

**Tests (Proof-aligned):**
1. Fire `dispatchBatch` with 3 agents mocked to take 500ms each. Assert total time < 900ms (parallel), not 1500ms (sequential).
2. Assert all 3 `agent_dispatches` rows have `started_at` within 100ms of each other (tight concurrency).
3. Assert `completed_at` ordering reflects actual completion order (fastest mock finishes first).
4. Assert one failing subagent does not block the others (Promise.allSettled semantics).

**Acceptance:** 4 tests green. No change to existing single-agent dispatch tool behavior.

### R3.5 — `/api/ceo/dispatches` polling endpoint + hook
**Files:**
- `src/app/api/ceo/dispatches/route.ts` — new (GET, auth-gated via existing `createClient`)
- `src/hooks/useDispatchProgress.ts` — new (React hook wrapping polling)
- `src/hooks/useDispatchProgress.test.ts` — new

**Endpoint:**
- `GET /api/ceo/dispatches?requestId=…` → `{ dispatches: [...] }`
- Requires `auth.uid()` session; queries `agent_dispatches` for the user + requestId.
- 404 if requestId not owned by user (RLS does the real work; the 404 is a UX nicety).

**Hook:**
- `useDispatchProgress(requestId: string | null, isActive: boolean)` → `Record<agent, { status, startedAt, completedAt }>`.
- Polls every 300ms while `isActive && requestId && at-least-one-agent-not-terminal`.
- Stops polling when all agents terminal OR `isActive` flips false.
- Cleans up interval on unmount.

**Tests (hook, vitest + fake timers):**
1. Starts polling when active + requestId set.
2. Stops polling once all dispatches terminal.
3. Cleans up interval on unmount.

**Acceptance:** tests green. Endpoint curl-test smoke (during dev-server phase, not CI).

## Ledger Shape
Each task gets:
- `tower start R3.<n>` (acquires lock)
- One or more commits prefixed `[R3/3.<n>] type: what`
- `tower done R3.<n>` on completion
- If blocked: `tower block R3.<n> "reason"`

Final step: `tower handoff --stdin` with full decisions list, then push.
