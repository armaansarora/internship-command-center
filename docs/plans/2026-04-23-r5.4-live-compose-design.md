# R5.4 — Live-Compose Streaming + Pen-Glow Ink (Design)

**Date:** 2026-04-23
**Session:** autopilot, scope R5-only mini-phase
**Phase goal verbatim (R5 Intent):** "you watch a cover letter compose itself in real time, as if the character is writing it."

## Intent

Three tones compose in parallel. Tokens land visibly, one after another, in the Ready-to-Send cards. A pen-glow cursor sits at the live edge of each draft, brightening as tokens arrive and dimming between them. The user is not watching a loading spinner; they are watching writing happen. `prefers-reduced-motion` drops the pen animation without dropping the streamed text — the composition is still live, just calmer.

## Non-negotiables (from partner constraints)

1. Streamed via AI SDK v6 `streamText`.
2. All three tones stream in parallel — promise-based fan-out that does NOT block on the first tone completing.
3. Each stream emits ≥10 token deltas before finishing (proof-asserted).
4. Three streams start within 100ms of each other (proof-asserted).
5. Final streamed text equivalent to non-streamed generation for the same prompt (proof-asserted — deterministic via mock, LLMs aren't seed-stable enough in prod for byte-equivalence).
6. Pen-glow cursor brightens on active stream, dims between tokens.
7. `prefers-reduced-motion` skips the pen animation but KEEPS the streaming text.

## Architecture

```
Client (LiveComposePanel)
  └── Fan-out: Promise.all([streamTone('formal'), streamTone('conversational'), streamTone('bold')])
        ├── POST /api/writing-room/compose-stream { tone: 'formal', ... }
        ├── POST /api/writing-room/compose-stream { tone: 'conversational', ... }
        └── POST /api/writing-room/compose-stream { tone: 'bold', ... }
              └── each: streamText({ model, system: toneSystemPrompt, prompt })
                  → response.toTextStreamResponse() (raw text/plain stream)
```

### Server

**New file:** `src/lib/ai/structured/cover-letter-stream.ts`

- Exports `streamCoverLetterProse({ userId, companyName, role, tone, jobDescription?, companyResearch? })` → `{ textStream: AsyncIterable<string>; fullText: Promise<string>; }`
- Uses same system prompt (`getToneSystemPrompt(tone)`) from `cover-letter.ts` so the three-tone divergence invariant still holds at the streaming layer.
- Uses `streamText({ model, system, prompt })` from AI SDK v6.
- Returns both the async iterable of text deltas AND a promise that resolves to the final assembled text, so callers can choose their integration depth.
- Telemetry via `recordAgentRun({ action: "structured.cover_letter.stream", ... })` on finish.

**New route:** `src/app/api/writing-room/compose-stream/route.ts`

- POST, auth-gated via `getUser()`.
- Body: `{ tone: 'formal' | 'conversational' | 'bold', companyName, role, jobDescription?, companyResearch? }` (Zod-validated).
- Returns `text/plain` token stream via `streamText(...).toTextStreamResponse()` (AI SDK v6 native — simplest matching-body format for per-tone parallel calls).
- 503 when AI provider not provisioned (matches the R5.1 pattern for graceful env-missing paths).

### Client

**New component:** `src/components/floor-5/live-compose/LiveComposePanel.tsx`

Props:
```ts
interface LiveComposePanelProps {
  companyName: string;
  role: string;
  jobDescription?: string;
  companyResearch?: string;
  onComplete?: (texts: Record<ToneKey, string>) => void;
}
```

Behavior:
- On mount, fires 3 parallel `fetch` POSTs to `/api/writing-room/compose-stream`.
- Each response body is consumed via `response.body!.getReader()`, decoded with `TextDecoder`, accumulated into per-tone `useState<string>`.
- Each decoded chunk triggers a "token event" — the pen-glow for that tone brightens, sets a 250ms timer to dim.
- When all three streams resolve, calls `onComplete(texts)`.
- Failure on one tone does not cancel the other two.

**New sub-component:** `src/components/floor-5/live-compose/PenGlowCursor.tsx`

- Small absolute-positioned SVG/CSS element rendered at the live edge of each card.
- `data-active="true"` when a token just arrived → CSS transition brightens `box-shadow` + `opacity`.
- Auto-dims after 250ms idle via internal `setTimeout`.
- `@media (prefers-reduced-motion: reduce)` → `display: none` (the streaming text remains).

### prefers-reduced-motion path

Detected via `useMediaQuery("(prefers-reduced-motion: reduce)")` hook (or inline `window.matchMedia`). When true:
- `<PenGlowCursor>` is not rendered at all.
- Streaming text still flows in — no animation on the text itself (tokens appear as they arrive, which is naturally progressive and not a transition).

## Proof tests (required)

**New file:** `src/lib/ai/structured/__tests__/live-compose.proof.test.ts`

Three assertions, all via mocked `streamText`:

1. **Parallel start within 100ms:**
   - Mock `streamText` to record `Date.now()` at call time and return a controlled async iterable.
   - Fire three `streamCoverLetterProse` calls via `Promise.all` (mirrors what the client will do).
   - Assert `max(startTimes) - min(startTimes) < 100`.

2. **Each stream emits ≥10 token deltas:**
   - Mock `streamText` to return a 15-chunk async iterable.
   - Run all three tones.
   - Assert each received chunk count ≥ 10.

3. **Streamed full-text equivalent to non-streamed generation:**
   - Mock `streamText` to emit `["Dear", " ", "Hiring", " ", "Team", ... 'formal' output]` chunks.
   - Mock `generateText` (non-stream path) to return the same assembled string wrapped in the expected structure.
   - Assert `await streamCoverLetterProse(input).fullText === generateStructuredCoverLetter(input).markdown` (after structural normalization — both are prose of the same content).

**Component test:** `src/components/floor-5/live-compose/LiveComposePanel.test.tsx`

- Mocks global `fetch` to return stubbed streams.
- SSR renders the panel (following the R8 pattern using `renderToStaticMarkup` since `@testing-library/react` isn't installed).
- Asserts: three live-compose cards present, each with `aria-live="polite"`, each with a PenGlowCursor element (SSR snapshot check).
- Assert `prefers-reduced-motion` honored via `matchMedia` mock → PenGlowCursor omitted, streaming text path still renders.

## Scope: in / out

**In:**
- Server util + route.
- Client panel + pen cursor + reduced-motion branch.
- Proof tests (3 assertions above) + component test (2 branches).

**Out (future wiring):**
- Deep integration into `WritingRoomClient` generation flow (the CMO tool's DB persist path stays unchanged in this phase). The panel ships as a standalone reusable surface; wiring into the CMO-tool generation flow is a post-R5.4 polish job.
- DB persistence from the stream endpoint. Persistence continues to live in the existing `generateThreeTonesTool` flow. The new stream endpoint is a UI-side renderer, not a DB writer. This is deliberate — keeps R5.4 surgical.

## Risks

- **LLM nondeterminism** — addressed by mocking `streamText`/`generateText` in the equivalence test.
- **Provider/Gateway mock shape** — mirrored from `three-tone-divergence.proof.test.ts`.
- **Streaming response body parsing** — Next.js App Router supports `ReadableStream` response bodies natively (already used by concierge route); no framework surprises expected.

None — every decision here sits within R5's Brief and the partner's explicit constraints.
