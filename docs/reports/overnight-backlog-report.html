⚠ Why the DB migration is committed-and-ready, not applied to prod

The directive's DB rail — "apply on a Supabase dev branch first, test up & down, then promote to production, then re-run the advisor" — is not mechanically reachable in this unattended environment. There is no SUPABASE_ACCESS_TOKEN (so no Management API → no branch create/promote, no advisor run, no Auth config), no psql, no Docker, and no Postgres driver; the direct DB is IPv6-only and the Supabase MCP needs interactive OAuth. This repo's actual migration mechanism is exactly this: a numbered SQL file in src/db/migrations/ that the owner applies via the Supabase SQL editor (see drizzle.config.ts). Blind-applying DDL to the live production DB with no advisor to verify and no dev-branch safety net is precisely the unverifiable risk the rollback doctrine forbids — so the migration is delivered one-paste-ready instead, with rollback inline.

Migration 0041 — up & down

All statements idempotent + reversible; the full rollback is inline in the file. The owner applies the UP in the Supabase SQL editor; the DOWN restores prior state with no data change.

-- UP (excerpt — full file: src/db/migrations/0041_security_advisor_rpc_hardening.sql)
-- handle_new_user: trigger-only → revoke direct EXECUTE
REVOKE EXECUTE ON FUNCTION public.handle_new_user() FROM anon, authenticated, PUBLIC;

-- consume_ai_call_quota: service-role only → re-assert revoke
REVOKE EXECUTE ON FUNCTION public.consume_ai_call_quota(uuid, integer) FROM anon, authenticated, PUBLIC;

-- bump_match_rate_limit: keep authenticated grant, add auth.uid() guard
CREATE OR REPLACE FUNCTION bump_match_rate_limit(p_user_id uuid, p_bucket timestamptz, p_limit int)
  ... SECURITY DEFINER SET search_path = public AS $$
  BEGIN
    IF p_user_id IS NULL OR p_user_id <> auth.uid() THEN RAISE EXCEPTION '...'; END IF;
    ... -- original upsert body unchanged
  END; $$;
GRANT EXECUTE ON FUNCTION bump_match_rate_limit(uuid, timestamptz, int) TO authenticated;
