## Executive Summary [3-5 sentences]
ArtLab has a serious surface-contract problem: Telegram, CLI, MCP, and docs describe one coherent engine, but several non-Telegram paths only write intermediate files or status text and never reach the same state-machine actions. The most damaging issues are broken run IDs between CLI produce and daemon queueing, inert CLI gate responses, an unimplemented cancel command that exits successfully, and SDK generate jobs that can seed character work without a character identity. The integrity layer has good raw materials, especially schema hardening, canonical roleSlug guards, and daemon lock handling, but observability and recovery have false-green paths (`health` exits 0 while daemon is down, queue dequeue can drop jobs after unlink). Memory learning is partially implemented but does not yet reliably learn from rejected concepts or route promotion wins into the SDK brain scope.

## Critical [data loss / broken contracts / silent failures / security]

1. **CLI `produce` returns a run ID that the daemon throws away.**
   - Evidence: `src/lib/artlab/cli/produce.ts:28-37` creates `runId`, writes `produce-${runId}.json`, but the payload only stores `request`, `sourceSurface`, and `createdAt`; it then reports `Queued run ${runId}`.
   - Evidence: `src/lib/artlab/daemon/cli-inbox-bridge.ts:51-61` drains `produce-*` intents, creates a new `randomUUID()`, and enqueues that new ID instead of preserving the CLI ID.
   - Why matters: a user who runs `artlab produce ...` gets an ID that `status`, `answer`, `continue`, and `cancel` cannot reliably target; the control plane starts with a broken contract.
   - Fix sketch: include `runId` in the produce intent payload and make the bridge preserve it, or stop returning a run ID until the daemon has created canonical state; add a produce -> bridge -> status regression test.

2. **CLI `answer` and `continue` are recorded but not consumed by the worker/state machine.**
   - Evidence: `src/lib/artlab/cli/answer.ts:22-37` and `src/lib/artlab/cli/continue.ts:22-34` write intent files under `.artlab/engine/inbox/cli`.
   - Evidence: `src/lib/artlab/daemon/cli-inbox-bridge.ts:65-79` moves those intents into `runs/<runId>/cli-inbox/`, while `src/lib/artlab/cli/run-worker.ts:41-84` only reads state/queue-entry and runs deterministic transitions.
   - Evidence: `src/lib/artlab/orchestrator/deterministic.ts:47-53` stops at `brief-review`, `concept-review`, and `final-review` with `awaiting-human-gate`; the CLI inbox files do not advance those gates.
   - Why matters: the docs call CLI a Telegram mirror (`docs/artlab/ENGINE.md:48-54`, `docs/artlab/OPERATIONS.md:26-31`), but the mirror can acknowledge an answer without applying it.
   - Fix sketch: bridge CLI answers into the same advance functions Telegram uses (`advanceConceptApproval`, brief adjust/approve, final approval), then enqueue the run; delete or fail unknown stale inbox files.

3. **`artlab cancel` is a successful stub despite cancellation being a safety property.**
   - Evidence: `scripts/artlab.ts:47-50` implements `stub()` that prints `fills in during Phase 1-3 implementation` and returns `0`; `scripts/artlab.ts:136-137` maps `cancel` to that stub.
   - Evidence: probe `npm run artlab -- cancel --help` exited `0` and printed `artlab cancel: stub — fills in during Phase 1-3 implementation`.
   - Evidence: docs promise `artlab cancel <runId>` writes cancel intent (`docs/artlab/ENGINE.md:54`) and that cancellation is honest (`docs/artlab/ENGINE.md:68-72`).
   - Why matters: operators can believe a paid generation was cancelled while the engine did nothing, which is both spend-risk and trust-risk.
   - Fix sketch: implement cancel as a real intent/state transition with SIGTERM, lease release, and refund handling; until then, return non-zero with a hard error instead of a stub.

4. **MCP `artlab/generate` character jobs can enter the engine without a character identity.**
   - Evidence: `src/lib/artlab/sdk/mcp/tools.ts:271-293` defines generate input with `kind`, `description`, `referenceImageUrl`, `anchorPackId`, `priority`, and `requesterAgent`, but no `characterId`.
   - Evidence: `src/lib/artlab/daemon/sdk-poller.ts:164-172` seeds `run-state.json` with `runId`, `assetType`, `phase`, `request`, and `sourceSurface`, but no `characterId`.
   - Evidence: `src/lib/artlab/cli/run-worker.ts:41-60` only calls `routeRequest` when no state exists; SDK poller has already written state. `src/lib/artlab/runners/brief-runner.ts:155-167` then throws when `input.characterId` is missing, and `src/lib/artlab/runners/brief-runner.ts:297-306` turns that into `no-character-match`.
   - Why matters: an agent-native MCP request for a character can be accepted and queued, then deterministically block at brief composition even if the description named a character.
   - Fix sketch: add explicit `characterId`/canon ID to generate, or run the router before seeding SDK state and persist the resolved `characterId`; merge validated `brainHint` identity into state before the worker starts.

5. **Queue dequeue can lose a job if runner spawn or registration fails after unlink.**
   - Evidence: `src/lib/artlab/queue/queue.ts:70-75` selects the next queue file, unlinks it immediately, then returns the entry.
   - Evidence: `src/lib/artlab/daemon/queue-processor.ts:24-35` calls `spawnRunner(next)` after dequeue and only then registers the child; there is no requeue path around a spawn throw or failed child setup.
   - Evidence: `src/lib/artlab/daemon/entry.ts:140-142` catches tick errors into daemon error logs, so the daemon can continue after the queue entry has already been removed.
   - Why matters: a transient spawn failure can drop paid work from the queue, contradicting the crash-recovery promise of no data loss (`docs/artlab/OPERATIONS.md:51-56`).
   - Fix sketch: atomically rename queue files to an `inflight/` claim before spawning, only delete after the child is registered, and requeue or mark failed with an explicit ledger on spawn errors.

## Important [UX / observability / schema drift / perf / doc drift]

1. **`artlab health` exits 0 while reporting daemon down.**
   - Evidence: `scripts/artlab.ts:110-115` always returns `0` after rendering the health snapshot.
   - Evidence: `src/lib/artlab/health/scanners/daemon-errors.ts:96-103` formats missing/stale heartbeat as `✗ Daemon down`; probe `npm run artlab -- health` printed `✗ Daemon down (no heartbeat)` and exited `0`.
   - Evidence: setup instructions use `npm run artlab -- health` as the verification step (`docs/artlab/OPERATIONS.md:13-15`).
   - Why matters: CI, onboarding, and operators can get a false green on the exact command docs tell them to trust.
   - Fix sketch: return non-zero for fatal health states in non-interactive mode; add `--json` or `--soft` if dashboards need a non-failing render.

2. **Health snapshot collects locks and daemon errors but the renderer hides the actionable parts.**
   - Evidence: `src/lib/artlab/health/snapshot.ts:47-60` includes `locks: scanLocks(...)` and `daemon: scanDaemonErrors(...)`.
   - Evidence: `src/lib/artlab/cli/ui/render.ts:105-153` renders daemon banner, spend, processes, leases, cleanup, and speed, but no lock table and no recent daemon-error section.
   - Evidence: `src/lib/artlab/health/scanners/locks.ts:14-30` already scans `.lock.*.json`, and docs say health covers locks (`docs/artlab/ENGINE.md:23-25`).
   - Why matters: the daemon can be blocked by stale/corrupt locks or repeating errors while the health UI omits the details needed for recovery.
   - Fix sketch: add `Locks` and `Recent daemon errors` blocks to `renderHealthView`; make stale/corrupt locks affect health severity and exit code.

3. **Telegram `reject` is parsed and advertised but does not abandon a run.**
   - Evidence: `src/lib/artlab/bot/reply-parser.ts:28-32` includes `{ type: "reject" }`, and `src/lib/artlab/bot/reply-parser.ts:40-60` maps `reject|archive` to that action.
   - Evidence: `src/lib/artlab/bot/bot-dispatcher.ts:454-475` handles gate replies by advancing only `approve-direction`; any other parsed action gets `gateReplyEcho`.
   - Evidence: templates tell the user `reject` abandons the run (`src/lib/artlab/bot/message-templates.ts:149` and `src/lib/artlab/bot/message-templates.ts:357-360`).
   - Why matters: the normal concept-review escape hatch is non-actionable; users trying to stop bad work get an echo instead of state change or memory learning.
   - Fix sketch: handle `parsed.action.type === "reject"` by marking the run cancelled/rejected, appending rejection memory with lane/reason when available, and sending a terminal acknowledgement.

   - Why matters: the system advertises accumulated taste, but some successful promotions are invisible to scoped SDK brain calls and rejected concepts do not reliably become future negative examples.
   - Fix sketch: write `source` on promotion wins, wire reject/revise-negative flows to `appendRejection`, and add a test that the next prompt receives non-empty recent wins/rejections.
