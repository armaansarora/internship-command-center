## Executive Summary

ArtLab's architecture is honest and well-instrumented in many places (SSRF allow-list on `referenceImageUrl`, atomic heartbeat writes, path-scoped auto-commit, poisoned-message quarantine), but the live system is shipping with three contract-violating regressions: the documented `artlab cancel <runId>` command is still a stub (no cancel intent written), the Gemini concept-critique call is hard-coded to a model Google retired (`gemini-3-pro-preview` → 404s in `daemon-errors.jsonl` from today), and the "memory learns from rejections" promise has no production write path — `appendRejection` is only ever called from test fixtures. There is broad canon ↔ docs role/floor drift across 6+ characters that means agents grounded in `docs/artlab/CHARACTER-PIPELINE.md` will get the wrong answer for every Phase-6 cast member. Daemon, state machine and reconciler core are solid; the friction is concentrated at the edges (docs, fallbacks, the cancel path, and unused-blocker plumbing) and in two skipped acceptance tests that mean the "≥40% faster than baseline" and "memory accumulates per promotion" properties have no CI coverage.

## Critical

### C1. `artlab cancel <runId>` is a stub — documented but does nothing
- **Evidence:** `scripts/artlab.ts:217`
  ```
  case "cancel":
    return stub("cancel", rest, io);
  ```
  `stub()` (scripts/artlab.ts:53-57) just prints `artlab cancel: stub — fills in during Phase 1-3 implementation` and returns exit 0.
- **Docs claim it works:** `docs/artlab/ENGINE.md:54` (`artlab cancel <runId> # write cancel intent`) and `docs/artlab/OPERATIONS.md:39` (Troubleshooting row tells operators to run `artlab cancel <runId>` when a run is stuck > 10 min).
- **Why it matters:** Safety property §3 ("Cancellation is honest") in `ENGINE.md:72` claims SIGTERM + grace + lease release + reservation refund. The CLI surface for that property is non-functional. The cancel-flow machinery in `src/lib/artlab/daemon/cancel-flow.ts` exists and works, but there is no way for a human to drop a cancel intent file from the CLI — they must reach for Telegram or edit `.artlab/engine/runs/<id>/cancel.json` by hand.
- **Fix sketch:** Wire `cancel` to a new `runCancelSubcommand` that writes `{ runId, requestedAt }` to `.artlab/engine/runs/<runId>/cancel.json` (the shape `cancel-flow.ts:27-32` parses), mirroring how `produce.ts` writes to the inbox.

### C2. Gemini concept-critique brain is hard-coded to a model Google retired
- **Evidence:** `src/lib/artlab/orchestrator/gemini-brain.ts:34` — `const DEFAULT_MODEL = "gemini-3-pro-preview";`
- **Live failure:** `.artlab/engine/daemon-errors.jsonl` (last two entries, today):
  ```
  concept-critique skipped: gemini-brain[critique-concept-board] failed after 3 attempts:
  gemini-brain gemini-3-pro-preview HTTP 404: {"error":{"code":404,"message":
  "This model models/gemini-3-pro-preview is no longer available..."}}
  ```
- **Why it matters:** Concept critique is the *only* multimodal QA gate on the concept board (see comment block at `src/lib/artlab/runners/concept-critique-blocker.ts:1-18`). Failure path silently blocks the run as `concept-critique-fallback` and continues. This is exactly the "concepts that should have been style-failed pass through" hole the helper was written to *surface*, but the surface only fires a blocker — the actual critique never ran. Tonight's promoted runs were not critiqued.
- **Fix sketch:** Change `DEFAULT_MODEL` to a Gemini 2.5 / 3-stable model (e.g. `gemini-2.5-flash` matching the image adapter at `providers/gemini-adapter.ts:32`); also remove the misleading comment at `gemini-brain.ts:9` claiming an automatic fallback to `gemini-3.5-flash` that doesn't exist for the critique path (the only fallback wrapper is in `concept-runner.ts:139-147` for the prompt-generation kind, not critique).

### C3. `appendRejection` has zero production callers — the rejection feedback loop is dead
- **Evidence:** `grep -rn appendRejection src/lib/artlab/` returns only `rejection-ledger.ts:22` (definition), `rejection-ledger.test.ts:12`, and `retrieve.test.ts:23`. No runner, no QA gate, no promotion code ever appends. Confirmed by `ls .artlab/engine/memory/` — no `style-rejections.jsonl` file exists on disk after 5+ recorded runs.
- **Docs lie:** `docs/artlab/CHARACTER-PIPELINE.md:51`: "Every rejected concept writes a `style-rejections.jsonl` entry. The LLM brain reads both via `getRelevantMemory` before generating the next character's prompts — so the cast accumulates lessons."
- **Why it matters:** `getRelevantMemory` (`memory/retrieve.ts:22`) dutifully reads rejections — but the file never contains anything, so the brain is half-blind. The "Memory feed-forward" advertised in the engine reference is wins-only in practice.
- **Fix sketch:** In `runners/strict-qa-runner.ts` (where QA failures are produced — around line 142-193) call `appendRejection` with the failing lane index + `qaFailureCodes` before transitioning to `style-failed`. Same for `concept-critique-blocker.ts:23-49` so a fallback also records a rejection entry, not just a blocker.

### C4. `concept-critique-fallback` blocker bypasses the state machine
- **Evidence:** `src/lib/artlab/types.ts:32-41` defines 8 blockers including `concept-critique-fallback`. But `src/lib/artlab/state/machine.ts:82-91` defines `BLOCKERS_FOR_TRANSITIONS` with only 7 — `concept-critique-fallback` is missing. `concept-critique-blocker.ts:33-38` then `writeRunStateSnapshot(runDir, { ...existing, blocker: "concept-critique-fallback" })` directly, bypassing `findBlockerTransition`.
- **Why it matters:** Spec §13 promises the reconciler is the single legal read path and BLOCKER_TRANSITIONS are the legal blocker-set transitions. This blocker exists in state files but `isLegalTransition(phase, phase, "concept-critique-fallback")` returns false (`machine.ts:106-111`) — you can write it, but the state machine can't recover from it or transition out of it via the documented blocker channel. Any future `isLegalTransition` validation gate will reject this blocker as illegal.
- **Fix sketch:** Either add `"concept-critique-fallback"` to `BLOCKERS_FOR_TRANSITIONS` at `machine.ts:82-90`, or rewrite `concept-critique-blocker.ts:33-37` to use the proper `findBlockerTransition` + `apply` path so the state machine remains the single source of truth.

### C5. Style-wins are keyed by `roleSlug`, not `header.id` — past wins won't match future canon lookups
- **Evidence:** Existing `.artlab/engine/memory/style-wins.jsonl` (2 entries today):
  ```
  {"characterId":"cno","promotedAt":"2026-05-25T16:37:44.919Z",...}
  {"characterId":"cno","promotedAt":"2026-05-26T00:08:00.750Z",...}
  ```
  Canon `header.id` for the CNO is `sol-navarro` (`docs/artlab/sdk/canon/characters/sol-navarro.yaml:4`); `roleSlug` is `cno` (line 6). Commit `ba8539db` explicitly switched future promotions to record `header.id` ("Record canon header.id as concept-board characterId via resolver"). `readStyleWins(...filter:{characterId:"sol-navarro"})` (`memory/style-ledger.ts:35`) does exact-match filtering and will MISS both legacy entries.
- **Why it matters:** When the next Sol regeneration calls `getRelevantMemory({ characterId: "sol-navarro" })`, the brain receives zero wins from the two prior promotions and starts cold. The whole point of the memory loop — accumulating lessons across the cast — is invisibly broken for every character promoted before the switch.
- **Fix sketch:** Either backfill the existing rows with `characterId: "sol-navarro"`, or make `readStyleWins` resolve the filter through `resolveCanonCharacter` so a query for either the slug or the canonical id finds both shapes.

### I2. `docs/artlab/ENGINE.md` describes a 10-phase machine + 7 blockers — code has 13 + 8
- **Evidence:** `ENGINE.md:31` lists 10 phases (`routed → generating-concepts → … → closed`) and `ENGINE.md:33` lists 7 blockers. `src/lib/artlab/types.ts:15-29` defines 13 phases (adds `briefing`, `brief-review`, `refining-concepts`). `types.ts:32-41` defines 8 blockers (adds `concept-critique-fallback`).
- **Why it matters:** New contributors and audit agents build a mental model from ENGINE.md that omits the brainstorm fork (`briefing`/`brief-review`) and the concept-refinement loop — both of which gate human responses. The brainstorm phases are actively visited by `state/machine.ts:51-58`.
- **Fix sketch:** Rewrite `ENGINE.md:29-35` to enumerate the 13 phases (with brainstorm fork noted explicitly) and the 8 blockers; both can be derived directly from the `as const` arrays in `types.ts`.

### I3. Default Claude model `claude-opus-4-5` is one generation behind in 5 files
- **Evidence:** `grep -rn ARTLAB_CLAUDE_MODEL src/lib/artlab/` shows the literal `claude-opus-4-5` default in `bot/commands.ts:207`, `runners/brief-runner.ts:34`, `runners/concept-runner.ts:129`, `runners/production-runner.ts:37`, `runners/promotion-runner.ts:62`. Current Anthropic Opus is `claude-opus-4-7` (system info).

### I4. `sdk-poller` writes `sourceSurface: "cli"` for MCP-origin runs — schema drift
- **Evidence:** `src/lib/artlab/daemon/sdk-poller.ts:164-172`:
  ```
  const state: ArtLabRunState = {
    runId: job.runId,
    ...
    sourceSurface: "cli",   // but job.source === "artlab-mcp"
  };
  ```
  `types.ts:83` declares `sourceSurface: z.enum(["telegram", "cli", "daemon-resume", "migration"]).optional()` — no `"artlab-mcp"` value exists. The queue spec immediately below correctly uses `sourceSurface: "artlab-mcp"` (line 191) but the run-state lies.
- **Why it matters:** `/status` and any observability filtering by `sourceSurface` will mis-attribute every MCP-originated run as a CLI run. The two surfaces are operationally different (CLI implies an interactive human; MCP implies an agent).
- **Fix sketch:** Add `"artlab-mcp"` to the `sourceSurface` enum at `types.ts:83` and use it at `sdk-poller.ts:171`.

### I5. Telegram identity-drop is silent — operator has no signal when Keychain is misconfigured
- **Evidence:** `src/lib/artlab/bot/identity.ts:13-17` returns `false` when the Keychain entry is missing OR the chat.id doesn't match; `bot-dispatcher.ts` then drops with `{ type: "dropped", reason: "unauthorized" }`. Neither the dispatcher nor the poller logs unauthorized drops to `daemon-errors.jsonl`. `OPERATIONS.md:36` documents "Telegram messages silently dropped" as a symptom but the only diagnostic is "verify the Keychain entry."
- **Why it matters:** A bricked Keychain access path looks identical to "no messages today" to the operator. Spec §6 ("Identity check (silent drop unauthorized Telegram chat.id)") justifies silence per *update* — but does not justify zero observability for the case where ALL messages are getting dropped because `authorizedChatId()` returns null.
- **Fix sketch:** On the first drop per daemon boot (state-machine style flag), `recordDaemonError("identity-guard", "all telegram messages dropping — tower-artlab-chat-id Keychain entry missing or unreadable")`. Subsequent drops stay silent.

### I6. `phase-5-acceptance.test.ts` and `cast-push-accumulation.test.ts` are `describe.skip` — both acceptance gates are uncovered
- **Evidence:** `src/lib/artlab/speed/phase-5-acceptance.test.ts:13` — `describe.skip("Phase 5 acceptance — Rafe-rerun ≥ 40% faster than baseline", …)`. `src/lib/artlab/memory/cast-push-accumulation.test.ts:6` — `describe.skip("Phase 6 memory accumulation (manual — run after each character)", …)`.
- **Why it matters:** The OPERATIONS.md "Phase 5 promise: ≥ 40% faster than `phase-4-rafe-baseline`" (`OPERATIONS.md:60`) has no automated proof. Same for the per-promotion memory growth that the cast-push runbook relies on. The skip comments justify *real-money guards*, but the result is two acceptance criteria that no CI run will surface a regression for.
- **Fix sketch:** Split each into two suites — a `describe("…shape")` that asserts the schema/structure with cheap fixtures (always runs) plus the existing `describe.skip("…live spend")` block. The structure test would have caught C5 above.

### I7. `improvementPercent: 0` ambiguity in the health speed summary
- **Evidence:** `src/lib/artlab/health/snapshot.ts:39` — `const baselineMs = baseline?.wallClockMs ?? median;` then line 41 — `improvementPercent = Math.round(((baselineMs - median) / baselineMs) * 100)`. When baseline is absent (`undefined`), `baselineMs === median`, so `improvementPercent` is exactly 0.
- **Why it matters:** The `/health` speed block renders `0%` *both* when no baseline has been recorded and when speed actually didn't improve — operator cannot tell the two apart. After a fresh checkout this looks like a regression.
- **Fix sketch:** Return `improvementPercent: undefined` (and have the renderer print "no baseline") when `baseline` is undefined, rather than collapsing the missing-baseline state into "0%".

### I8. Promotion auto-pushes to git remote by default — not documented in OPERATIONS.md
- **Evidence:** `src/lib/artlab/runners/promotion-runner.ts:303-318` calls `autoCommitPromotion(... skipPush: process.env.ARTLAB_AUTO_PUSH === "off")`. Default behavior: AUTO-COMMIT and AUTO-PUSH are on. `OPERATIONS.md` mentions byte-diff CI but never that the daemon will `git push` on its own.
- **Why it matters:** Operator running ArtLab on a feature branch may have promotion commits sail to `origin/<their-branch>` without intending to. Combined with the C1 cancel stub, the only way to stop an in-flight promotion is to kill the daemon — by which time the push has already happened.
- **Fix sketch:** Document the env flags in `OPERATIONS.md` "Daily use" section, AND default `skipPush: true` until the operator opts in via `ARTLAB_AUTO_PUSH=on`. Path-scoping helps but doesn't change the surprise.

## Minor

### M1. `daemon-heartbeat.json` is written *before* any work each tick
- **Evidence:** `src/lib/artlab/daemon/entry.ts:144-145` — `runDaemonOnce` calls `writeHeartbeat` as line 1 of the tick. Subsequent steps are wrapped in `runStep` which catches all errors. Result: every step can be failing and the heartbeat stays fresh — `daemon-heartbeat.json` proves liveness but not progress.
- **Fix:** Tag the heartbeat with the last successful step and `lastFailureAt` per source, so `formatDaemonBanner` can distinguish "alive and ticking" from "alive but every step throws."

### M2. `GEMINI_API_KEY.startsWith("__")` placeholder rejection is undocumented convention
- **Evidence:** Same six-line guard appears in `bot/commands.ts:208`, `brief-runner.ts:35`, `concept-runner.ts:61`, `production-runner.ts:79`, `promotion-runner.ts:63`. No comment or README anywhere explains why `__` is the sentinel.
- **Fix:** Extract `function isPlaceholderApiKey(v?: string): boolean` next to `geminiKeyFromEnv` (concept-runner.ts:60) and reference it. One change site, one comment.

### M3. `daemon-run.ts` swallows `.env.local` parse failures silently
- **Evidence:** `src/lib/artlab/cli/daemon-run.ts:101` — `} catch { /* swallow — daemon should boot even if .env.local is malformed */ }`. A typo in `.env.local` boots a daemon with missing env, and the operator only finds out via downstream provider 401s.
- **Fix:** Print `daemon: .env.local parse failed: <message> — booting without overrides` to stderr; don't crash, but don't be invisible.

### M4. CLAUDE.md still references `docs/CHARACTER-PROMPTS.md` as the source of character voice prompts
- **Evidence:** `CLAUDE.md:35` — "Character voice prompts: `docs/CHARACTER-PROMPTS.md`". The Floor Directory itself uses CRO/CNO/CIO/CMO etc. but legacy CHARACTER-PROMPTS.md is in `docs/legacy/` per `CHARACTER-PIPELINE.md:3` ("Consolidated from six legacy docs (now in `docs/legacy/`)"). New canonical home is `docs/artlab/CHARACTER-PIPELINE.md`.
- **Fix:** Update the pointer in CLAUDE.md so agents reading it don't chase a legacy file.

### M5. `recordDaemonError` truncates `message` to 200 chars on read but stores the full stack on write
- **Evidence:** `recordDaemonError` (`daemon/entry.ts:127-138`) writes the full stack. `scanDaemonErrors` (`health/scanners/daemon-errors.ts:69`) slices the message to 200 chars when surfacing the *last* error in `/health`, but the full multi-line stack is still in the file (visible in tonight's 4 telegram-poller `TypeError: fetch failed` entries with 10+ lines each).
- **Fix:** Truncate at write time too (cap to ~1KB per entry); over time the 256KB tail window misses fewer events.

## Surprising wins

### W1. `referenceImageUrl` SSRF defense is *actually defensive*
- `src/lib/artlab/sdk/mcp/tools.ts:21-84` — HTTPS-only, explicit host allow-list of 4 production hosts, *plus* an explicit deny set (`127.0.0.1`, `169.254.169.254`, `metadata.google.internal`), *plus* RFC1918 + link-local + loopback IPv4 ranges parsed by hand. This is markedly above the "just allow https" bar most projects ship.

### W2. `resolveCanonCharacter` actually logs roleSlug fallbacks
- `src/lib/artlab/sdk/canon/resolve-character.ts:44-50` emits a single INFO log per fallback via `options.log` — wired through to the brain log in `character-master/index.ts:69` (`console.info(JSON.stringify(entry))`). Observability for slug/id drift is real, not aspirational.

### W3. Poisoned inbox messages get quarantined to `.bad/`, not retried
- `src/lib/artlab/daemon/sdk-poller.ts:133-141` — `quarantine` moves the bad file to `inbox/sdk/.bad/<ts>-<filename>` and even catches errors from the quarantine itself before logging them. The poller can't get stuck on a single bad file.

### W4. Promotion auto-commit is path-scoped
- `src/lib/artlab/runners/promotion-runner.ts:296-318` — comment block at 296-300 explicitly notes "path-scoped staging is enforced inside autoCommitPromotion — the daemon can ONLY stage files under public/art/ or the generated manifest JSON." Defense in depth: even if a runner bug stages bad paths, the auto-commit refuses.

### W5. Heartbeat write is atomic
- `src/lib/artlab/daemon/entry.ts:114-125` — write to `daemon-heartbeat.json.tmp.<pid>.<ts>` then `renameSync(tmp, path)`. Same pattern in `telegram-poller.ts:28-34` for the offset file. Crash mid-write leaves the previous good copy untouched. (Worth pairing with M1: atomicity is great, but the *meaning* of the heartbeat still needs sharpening.)
