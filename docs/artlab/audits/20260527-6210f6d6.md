## Critical [data loss / broken contracts / silent failures / security]

### 1. CLI cancellation is advertised as honest but is a zero-exit stub.

Evidence:
- Docs advertise `artlab cancel <runId>` as writing a cancel intent (`docs/artlab/ENGINE.md:54`) and list cancellation honesty as a safety property with SIGTERM, lease release, and refund (`docs/artlab/ENGINE.md:72`).
- The CLI help repeats `artlab cancel <runId> cancel a run with refund` (`scripts/artlab.ts:36`).
- The implementation routes `cancel` to `stub("cancel", rest, io)` (`scripts/artlab.ts:136`), and the shared `stub` prints a message and returns success (`scripts/artlab.ts:47`, `scripts/artlab.ts:50`).
- Probe: `npm run artlab -- cancel` exited `0` and printed `artlab cancel: stub — fills in during Phase 1-3 implementation`.

Why matters: an operator or agent can believe a spend-bearing run was cancelled while no cancel intent, SIGTERM, lease release, or refund path ran. This is a broken safety contract, not just a missing convenience command.

Fix sketch: replace the stub with `runCancelSubcommand`, require a run id, validate the run exists, write the same `inbox/cancel-*.json` intent shape used by Telegram, and return non-zero on missing/unknown run ids. Add a CLI test that asserts no stub text, intent file creation, and non-zero invalid usage.
