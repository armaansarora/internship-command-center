## Executive Summary

The static surfaces of ArtLab are remarkably clean — strict Zod schemas with path-traversal + SSRF guards, single-instance daemon lock, atomic state writes, log rotation, byte-protected promoted art. But the **live runtime is in a partial-outage state right now**: the Gemini brain is hard-failing on every run because `gemini-3-pro-preview` (the hardcoded default) was deprecated and returns 404, the concept-critique blocker the team shipped 2 commits ago is being silently overwritten by the orchestrator, and the `characterId` schema fix on commit ba8539db only patched one downstream artifact while five others still carry the stale role-slug. The `cancel` CLI advertised in docs/OPERATIONS.md is a stub returning exit 0, and the "memory feed-forward" promise in CHARACTER-PIPELINE.md is fiction — `appendRejection` has no production caller. Five active runs are visibly stuck in concept-review with no blocker rendered, exactly as the silent-fallback failure mode predicts.

## Critical

### C1. `gemini-3-pro-preview` model is dead — every brain call 404s in production

- **Evidence:** `src/lib/artlab/orchestrator/gemini-brain.ts:34` — `const DEFAULT_MODEL = "gemini-3-pro-preview";` and `.artlab/engine/daemon-errors.jsonl` (last entry 2026-05-27T22:34:46Z): `"gemini-brain[critique-concept-board] failed after 3 attempts: gemini-brain gemini-3-pro-preview HTTP 404: This model models/gemini-3-pro-preview is no longer available."` Same error logged at 22:15:19. Historical decisions in `.artlab/engine/memory/decision-log.jsonl` used `model: "gemini-3.5-flash"` successfully (May 25).
- **Why it matters:** The brain is the system. Critique, recommendation, brief composition, concept-prompt refinement, repair plans — all route through this default model when Anthropic key isn't set or fails. Every brain call wastes 3 retry attempts then falls back to deterministic defaults (lane 3, canonical brief, no critique). No operator alarm; only daemon-errors.jsonl reflects it.
- **Fix sketch:** Move `DEFAULT_MODEL` to env-driven (`ARTLAB_GEMINI_BRAIN_MODEL`) with a working default like `gemini-2.5-pro` or `gemini-3-pro` (whichever Google currently exposes); update the comment at gemini-brain.ts:7-9 which falsely claims auto-fallback to `gemini-3.5-flash`.

### C2. Concept-critique-fallback blocker is silently overwritten by the orchestrator

- **Evidence:** Sequence triggers a race:
  1. `src/lib/artlab/orchestrator/deterministic.ts:44` reads run-state into local `state`.
  2. Runner executes; inside, `src/lib/artlab/runners/concept-runner.ts:447` calls `writeConceptCritiqueFallbackBlocker` which writes `blocker: "concept-critique-fallback"` to run-state.
  3. Runner returns `status: "ok"` (concept-runner.ts:458-461 — there is no `blockerHint` field).
  4. `deterministic.ts:89-101` walks the auto-transition path using the stale `state` (no blocker), and `src/lib/artlab/state/machine.ts:19-26 patch()` constructs a new state with `blocker: undefined`.
  5. Line 101 `writeRunStateSnapshot(input.runDir, stamped);` — the blocker is gone.
- **Live evidence:** `.artlab/engine/runs/44901b6f-…/run-state.json` shows phase `concept-review`, NO blocker field, while `.artlab/engine/daemon-errors.jsonl` recorded the same run's `concept-critique-fallback` source. Five runs are stuck in concept-review/final-review with this pattern (`npm run artlab -- status`).
- **Why it matters:** The recent commit `2605026d "Surface concept-critique fallback as a loud blocker"` does not actually surface a blocker — it logs to daemon-errors but the loud `⚠️ blocked: concept-critique-fallback` line in `/status` never renders. Quality-gate failures are invisible to the operator.
- **Fix sketch:** The runner must NOT write to run-state out-of-band. Add `concept-critique-fallback` to `ArtLabRunnerResult.blockerHint` typing and return it from the concept-runner; let `deterministic.ts:80-86` (the `result.status === "failed"` branch) own the blocker write. Alternatively, change `patch()` in `state/machine.ts:19` to preserve `state.blocker` unless the transition explicitly clears it.

### C4. `artlab cancel` advertised in docs and HELP is a stub returning exit 0

### C5. `style-rejections.jsonl` is read but never written — memory-feed-forward is fiction

- **Fix sketch:** Wire `appendRejection` into (a) concept-refinement when feedback rejects lanes, (b) strict-QA-runner when repairs are required, (c) the bot's `/cancel` / rejection paths. Align `RejectionEntry` shape with brain-agent reader shape.

- **Fix sketch:** On daemon abnormal termination, ensure either heartbeat is invalidated (delete it) or include a `lockHolderPid` and use the conjunction; document the "fresh heartbeat + dead pid" condition in the doctor output (it's a crash-during-tick signal, not "unknown state").

## Surprising wins

1. **Asset Pack manifest schema is hardened against path-traversal + injection.** `src/lib/artlab/sdk/asset-pack/manifest.schema.ts:39-66 isPathSafeAgainstTraversal` rejects empty, NUL, backslash, percent-encoded, leading `/`, leading `~`, drive prefix, post-normalize-drift, and `..` segments. Combined with strict `Sha256Hex` regex and explicit `JSON.stringify`-quote in `integration-snippet.ts:29-31` for code-injection defence — this is genuinely senior-engineer work. A malicious pack manifest cannot escape `public/`, can't inject HTML through `altText`, can't break out of `gsap.fromTo(…)` via `easing`.

2. **Daemon single-instance lock is correct.** `src/lib/artlab/daemon/daemon-lock.ts` uses atomic `open(path, "wx")` for the lock file, reclaims stale locks only when PID is genuinely dead (`process.kill(pid, 0)`), distinguishes EPERM from ESRCH (so root-owned daemons aren't mistakenly nuked), and the release closure is idempotent. The whole file is 127 lines including comments — tight and correct.

3. **MCP `generate` input validation is thoughtful.** `src/lib/artlab/sdk/mcp/tools.ts:271-294` validates description min/max chars, HTTPS-only + host-allow-list for reference image URLs (SSRF closure), `PackIdSchema` reused for `anchorPackId` (path-traversal closure), and strict enum on `priority`. Closes attack surface a 1-day shop would miss.

4. **`brief-runner` explicitly refuses the "silent lobby fallback".** Lines 159-167 throw rather than fall through to a wrong character — comment names the past failure: "the previous implementation had — it confused users into thinking their `make Sol Navarro` request had been routed to the lobby". This is the right pattern; pity it isn't applied at concept-critique.

5. **Doctor command is what a session-readiness check should be.** `npm run artlab -- doctor` runs five checks (settings, daemon, canon, promoted, sdk-inbox) and prints exactly one line each. Found the dead daemon and confirmed canon load in one invocation. Discoverable from SKILL.md fallback section.
